# ユーザー登録フロー ゴール検証質問

## ユーザーストーリー

As a 登録を希望する利用者,
I want to 登録申請からメールアドレス本人確認とパスワード登録を経て利用登録を完了したい,
So that 承認された利用者としてサービスを利用開始できる.

## 確定している仕様

- ユーザー登録フローは、登録申請フェーズ、メールアドレス確認フェーズ、パスワード登録フェーズで構成される。
- ユーザーは登録申請できる。
- 登録申請後、システムは「ユーザーの登録が申請された」状態を扱う。
- 登録申請の審査結果が承認の場合、登録申請は受け付けられたものとして扱う。
- 「登録申請が受け付けられた」は「承認された」と同義として扱う。
- 承認後、メールアドレス本人確認フェーズへ進む。
- 却下の場合、ユーザーに通知してフローを終了する。
- 承認後、メールアドレス本人確認が依頼される。
- ユーザーはメールアドレスが本人のものであることを確認する。
- メール本人確認が成功した場合のみ、パスワード登録フェーズへ進む。
- メール本人確認で確認期限切れになった場合、タイムアウトとしてフロー終了する。
- メール本人確認で確認トークン不正などの認証エラーになった場合、再度メール本人確認フローへ戻す。
- メール本人確認成功後、パスワード登録が依頼される。
- ユーザーはパスワードを登録できる。
- パスワードが正常な場合、ユーザー登録を完了する。
- パスワードが正常でない場合、エラーを返し、ユーザーは再入力できる。
- 登録完了後、ユーザーへ登録完了通知を送る。

## ゴール検証

### ゴールの粒度

- 現時点のゴールは、登録申請から登録完了までの一連の登録フロー全体を対象としている。
- 1 スプリントで完了可能かどうかは、後続の planning で実装タスク分割時に評価する。

### 独立性

- 登録申請、メールアドレス本人確認、パスワード登録、通知が一連の価値提供に含まれる。
- 他 PBI への依存は、現在の task 指示からは明示されていない。

### 検証可能性

- 正常系、却下ルート、メール本人確認タイムアウト、メール本人確認認証エラー、パスワード不正、登録完了通知のシナリオで検証する。

### スコープ境界

- 含むもの: 登録申請、申請審査結果による分岐、メールアドレス本人確認、パスワード登録、却下通知、本人確認依頼通知、パスワード登録依頼通知、登録完了通知。
- 含まないもの: 文言や画面表示の詳細、将来拡張を前提にした追加仕様、実装方式、ファイル構成、技術選定。

## 未確定の確認事項

1. 登録申請の承認 / 却下主体が未確定。
   - 何が未確定か: 登録申請の審査結果を誰または何が決定するか。
   - 回答で確定すること: scenario で扱う審査操作の主体と、登録申請フェーズの分岐前提。
